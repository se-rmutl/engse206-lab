# วิธีใช้ Semester Project Case ตลอดภาคเรียน

## หลักการ

อาจารย์กำหนดโจทย์มาตรฐาน 1 เรื่องต่อ 1 กลุ่มตั้งแต่ Week 1 แล้วใช้โจทย์เดิมต่อยอดจน Week 16 เพื่อให้ผู้เรียนเห็นความเชื่อมโยงของ artefact ทั้ง flow ไม่ใช่ทำใบงานแยกส่วน

## เหตุผลที่ใช้โจทย์มาตรฐาน

- ควบคุมขนาดและระดับความยากของงานให้ใกล้เคียงกัน
- ให้คะแนนเทียบกันได้อย่างเป็นธรรม
- ลดเวลาหาโจทย์ และเพิ่มเวลาฝึก requirements/design จริง
- ผู้สอนเตรียม role cards และข้อมูลเพิ่มสำหรับ stakeholder simulation ได้
- นักศึกษามี context เพียงพอสำหรับต่อยอด SRS, architecture, prototype และ detailed design

## กติกา

1. กลุ่มมีสมาชิก 3–4 คน เว้นแต่ผู้สอนกำหนดเป็นอย่างอื่น
2. ใช้ Case Card เดิมตลอดภาคเรียน
3. เก็บหลักฐานการตัดสินใจใน Git เช่น commit, worklog, meeting notes และ review comments
4. Requirement เปลี่ยนได้ แต่ต้องบันทึก change request, rationale และผลกระทบต่อเอกสาร/แบบออกแบบ
5. ไม่ใช้ข้อมูลจริงที่ระบุตัวบุคคลได้
6. ไม่บังคับทำโปรแกรมเสร็จ แต่ต้องมีเอกสารและแบบออกแบบที่ทีมพัฒนาทำต่อได้

## ขอบเขตที่พอดี

- ผู้ใช้หลัก 3–4 บทบาท
- ฟังก์ชันหลัก 8–12 ฟังก์ชัน
- โฟกัส workflow หลัก 1–2 workflow
- ใช้ข้อมูลจำลองหรือ anonymized data
- การเชื่อมต่อภายนอกอธิบายเป็น dependency ได้ แต่ไม่ต้อง implement

## ผลงานปลายภาค

1. Problem Brief และ Scope Statement
2. Stakeholder Map / System Context / Elicitation Evidence
3. Prioritized Requirements Backlog
4. SRS v1 และ Review/Revision Record
5. Traceability Matrix และ Change Record
6. Design Strategy + Conceptual Architecture + Rationale
7. User Flow / Wireframe หรือ Prototype + Usability Evidence
8. Detailed Design
9. Design Review Report / Improvement Backlog
10. Final Presentation และ Individual Reflection
