ตัวอย่างนี้ใช้กรณีศึกษา **Campus Resource Booking** เพื่อแสดงระดับรายละเอียดที่คาดหวังเมื่อจบ Week 1

## ไฟล์ในโฟลเดอร์

| ไฟล์ | จุดประสงค์ |
|---|---|
| [`problem-brief-v0.1.md`](problem-brief-v0.1.md) | ผลงานหลัก: problem, facts, pain points, stakeholder, goals, scope, starter requirements, NFR และ open questions |
| [`team-worklog.md`](team-worklog.md) | หลักฐานการมีส่วนร่วม การตัดสินใจ และ reflection ของทีม |

## การเชื่อมต่อไปยัง Week 2–4

```text
Week 1 Problem Brief v0.1
  ├─ Stakeholder เริ่มต้น ───────────────→ Week 2 Stakeholder Analysis
  ├─ Context / Scope / Constraints ─────→ Week 2 System Context & Scope
  ├─ Open Questions (OQ-ID) ────────────→ Week 3 Elicitation Plan & Interview Guide
  └─ Starter Requirements (UR-ID) ──────→ Week 4 Evidence / Negotiation / Candidates
```

## สิ่งที่ควรสังเกต

1. Problem statement ไม่เริ่มจาก “ต้องทำแอป”
2. Facts มีแหล่งที่มา ส่วน assumptions มีวิธีตรวจสอบ
3. Scope จำกัดให้เหมาะกับบริบทของรายวิชา
4. Starter requirement ยังเป็น Draft และมี evidence/reason
5. Open Questions ถูกออกแบบให้ใช้ต่อใน Week 3–4
6. มีการเปิดเผยการใช้ AI และไม่ใช้ข้อมูลส่วนบุคคลจริง
